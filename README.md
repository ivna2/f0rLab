# Fitness Club Service

Тема проекта: сервис управления фитнес-клубом.

Основные сущности:
- `Member` — участник клуба.
- `Trainer` — тренер.
- `Lesson` — занятие с тренером, датой и лимитом мест.
- `Subscription` — абонемент участника.
- `Booking` — запись участника на занятие.
- `AppUser` — пользователь безопасности с логином, email, хешем пароля, ролями и привязкой к участнику.
- `AdminNotification` — уведомление для администратора о действиях пользователей.

Что умеет сервис:
- CRUD для участников, тренеров, занятий, абонементов и записей.
- Регистрация пользователей и вход в систему.
- Ролевой доступ для `ADMIN` и `MEMBER`.
- Работа с кабинетом участника: просмотр профиля, ближайших записей, загрузка фото, перенос и отмена записи, смена пароля.
- Панель администратора: управление сущностями клуба, просмотр уведомлений, создание и удаление администраторов.

Бизнес-операции:
- оформление нового участника;
- запись участника на занятие;
- перенос записи на другой слот;
- продление абонемента;
- просмотр расписания участника;
- просмотр загрузки тренера.

База данных:
- основная БД — PostgreSQL;
- конфигурация БД вынесена в env-переменные;
- пароль БД не хранится в коде и должен передаваться через `DB_PASSWORD`;
- пример env находится в `.env.example`.

Ограничения и связи:
- уникальный email участника;
- уникальные login и email пользователя;
- уникальная пара `trainer_id + date_time` для занятий;
- уникальная пара `member_id + lesson_id` для записей;
- связи `Lesson -> Trainer`, `Subscription -> Member`, `Booking -> Member`, `Booking -> Lesson`, `AppUser -> Member`.

Безопасность:
- подключен Spring Security;
- включены Basic Auth и form login;
- включена CSRF-защита через cookie token;
- первый зарегистрированный пользователь получает роль `ADMIN`;
- все следующие пользователи получают роль `MEMBER`;
- пароль проверяется на сложность: длина 12-128, буквы в разных регистрах, цифра, спецсимвол, без пробелов и очевидных последовательностей;
- пользователи хранятся в таблицах БД, а не в коде или SQL-скриптах.

Тестовые данные:
- предметные данные не зашиваются в код фиксированными пользователями;
- после запуска можно создать свои сущности через UI или Postman-коллекцию;
- в проекте есть Postman-коллекция для проверки сценариев CRUD и бизнес-операций.

Коллекция запросов:
- Postman-коллекция: `requests/fitness-club.postman_collection.json`
- окружение: `requests/fitness-club.postman_environment.json`

Полезные точки входа:
- `GET /login` — страница входа;
- `GET /register` — страница регистрации;
- `GET /api/auth/csrf` — получить CSRF-токен для Postman/REST-клиентов;
- `POST /api/auth/register` — регистрация пользователя;
- `GET /api/auth/me` — информация о текущем пользователе.
